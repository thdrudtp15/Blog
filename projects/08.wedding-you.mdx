# 🚩 프로젝트 개요

5인 팀 프로젝트로 진행한 웨딩 플랜 플랫폼으로, **웨딩홀 견적서 관리 시스템**의 프론트엔드 및 백엔드 전체를 담당했습니다.

# 🔧 기술적 해결

## 1. 다중 이미지 업로드 처리

**문제 상황:**

-   견적서당 여러 장의 참고 이미지 업로드 필요
-   별도의 이미지 테이블을 생성하지 않고 간단하게 구현하고자 함

**해결 방법:**

-   업로드된 여러 이미지의 **파일 경로를 JSON 배열 형태의 문자열로 변환**하여 DB에 저장
-   `@Column(length=5000)` 설정으로 충분한 저장 공간 확보
-   조회 시 JSON 파싱을 통해 이미지 경로 배열로 복원하여 화면에 표시
-   이를 통해 별도의 조인 없이 간단하게 다중 이미지 관리 구현

---

## 2. 선호도 기반 카테고리 선택 시스템

사용자가 웨딩 관련 카테고리(드레스, 메이크업, 스튜디오 등)를 선호도 순으로 최대 3개까지 선택할 수 있는 기능 구현

**문제 상황:**

-   사용자가 최대 3개의 카테고리를 선호도 순서대로 선택
-   3개 선택 후 추가 선택 시, 가장 먼저 선택한 항목을 자동으로 제외
-   선택 순서를 시각적으로 표시 (1순위, 2순위, 3순위)

**해결 방법:**

-   배열의 길이가 3을 초과하면 **shift()** 메서드로 가장 오래된 항목 자동 제거
-   배열의 인덱스를 활용하여 선택 순서(1순위, 2순위, 3순위) 표시

---

## 3. 페이지네이션 구현

견적서 목록의 효율적인 조회를 위한 페이지네이션 기능 구현

**문제 상황:**

-   대량의 견적서 데이터를 한 번에 조회 시 성능 저하
-   전체 페이지 수 계산 로직 필요

**해결 방법:**

-   **LIMIT을**을 활용한 페이지 단위 데이터 조회
-   **Math.ceil()** 함수를 사용한 총 페이지 수를 계산

```java
int totalPages = (int) Math.ceil(totalCount / pageSize);
```

-   페이지 번호, 페이지 크기, 총 개수를 포함한 응답 데이터 구성

**결과:**

-   한 페이지당 일정 개수의 데이터만 조회하여 성능 최적화
-   프론트엔드에서 페이지 번호 버튼을 동적으로 생성할 수 있는 구조 제공

# 🤔 아쉬운 점 및 개선 방향

## 컴포넌트 구조화 부족

담당한 기능들을 **재사용 가능한 컴포넌트로 분리하지 못해** 코드가 길어지고 유지보수가 어려웠다. 특히 당시 Git 충돌에 대한 부담감으로 새로운 파일 생성을 꺼려했던 것이 컴포넌트 분리를 주저하게 만든 주요 원인이었다. 지금 생각해보면 충돌은 협업에서 자연스러운 과정이고, 적절한 브랜치 전략으로 충분히 관리 가능한 부분이었는데 당시에는 그 부분에 대한 이해가 부족했던 것 같다.
